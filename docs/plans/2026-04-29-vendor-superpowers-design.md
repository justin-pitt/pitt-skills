# Vendor superpowers — design

**Status:** approved 2026-04-29 (brainstorm phase complete)
**Target release:** v1.1.0
**Branch:** `feat/vendor-superpowers` (off main, post-M4 merge)

## Goal

Bake all 14 [obra/superpowers](https://github.com/obra/superpowers) skills into pitt-skills so coworkers (mostly Claude Code + Copilot users at CDW) get them automatically when they install pitt-skills via the universal install prompt — no follow-up "now also install superpowers" instruction needed. Justin's must-haves: `brainstorming`, `systematic-debugging`, `subagent-driven-development`. The rest come along to preserve cross-references.

## Approach

Hybrid — different harnesses get superpowers via different mechanisms:

- **Claude Code:** `settings.snippet.json` is extended to enable `superpowers@superpowers-dev` from the upstream marketplace (already registered there since M2). Users get fresh-from-upstream content automatically when they run `install.ps1` / `install.sh`.
- **Copilot CLI / VS Code Chat:** the 14 SKILL.md files are vendored into `plugins/pitt-skills/skills/`. `build.ps1` then generates `.github/instructions/<name>.instructions.md` for VS Code Chat and the `~/.copilot/skills/` symlink picks up the SKILL.md files directly for Copilot CLI.

This means Claude Code users get the live upstream and Copilot users get a versioned snapshot. The two can drift; a `scripts/sync-superpowers.ps1` helper keeps the snapshot easy to refresh.

## What gets vendored

All 14 skills from `obra/superpowers` skills/ at HEAD (currently SHA TBD when work begins; pin to whatever HEAD is at vendoring time):

```
brainstorming                       receiving-code-review
dispatching-parallel-agents         requesting-code-review
executing-plans                     subagent-driven-development
finishing-a-development-branch      systematic-debugging
test-driven-development             using-git-worktrees
using-superpowers                   verification-before-completion
writing-plans                       writing-skills
```

Each gets:

- `plugins/pitt-skills/skills/<name>/SKILL.md` — verbatim copy, but with `license: MIT` added to the frontmatter (Copilot CLI validator requires it; superpowers' own frontmatter omits it). Skill `name:` field stays as upstream wrote it.
- `plugins/pitt-skills/skills/<name>/UPSTREAM.md` — provenance record matching the M2 vendor format used in PR #4 (richer than `vendor-skill.ps1`'s default template). Fields: Repo, Path within repo, Commit SHA at vendoring, Original license, Vendored on, "My changes" with the `license:` addition note.

Cross-references (e.g., `subagent-driven-development` mentioning `superpowers:test-driven-development` in body text) are left as-is. Users with both pitt-skills and superpowers (Claude Code) resolve them naturally; Copilot users see them as text in the instruction file — harmless since Copilot has no Skill tool to invoke them anyway.

## What does NOT get vendored

- superpowers' `agents/`, `commands/`, `hooks/`, `tests/`, `scripts/` — Claude-Code-runtime-specific surfaces that don't translate to Copilot, and that Claude Code users get directly via the marketplace plugin
- superpowers' `docs/` and `assets/`
- superpowers' top-level `package.json`, `gemini-extension.json`, etc.

If a superpowers skill references files in those dirs from its SKILL.md body (e.g., a Bash script in `scripts/`), the reference is left in the vendored copy as text. Users in Copilot won't be able to invoke the script; users in Claude Code resolve the reference via the live marketplace plugin alongside.

## CI implications

The version-bump CI rule (added in M4 Task 30) will fire on this PR — 14 new SKILL.md files in `plugins/pitt-skills/skills/*/SKILL.md` require `plugin.json` to bump. Bumping to 1.1.0 satisfies the rule; no `[skip-bump]` needed.

`verify-build` will need to see all 14 generated instruction files committed — `build.ps1` regenerates them, so the dispatch's last step before commit must be `pwsh ./scripts/build.ps1` followed by staging the regenerated artifacts.

`pester` and `bats` are unaffected.

## sync-superpowers.ps1

Convenience wrapper for future refreshes. Pseudocode:

```powershell
[CmdletBinding()]
param([Parameter(Mandatory)] [string] $CommitSha, [switch] $Force)

$skills = @(
    'brainstorming','dispatching-parallel-agents','executing-plans',
    'finishing-a-development-branch','receiving-code-review','requesting-code-review',
    'subagent-driven-development','systematic-debugging','test-driven-development',
    'using-git-worktrees','using-superpowers','verification-before-completion',
    'writing-plans','writing-skills'
)

$tmp = Join-Path ([System.IO.Path]::GetTempPath()) "superpowers-$CommitSha"
git clone --depth=1 https://github.com/obra/superpowers $tmp
Set-Location $tmp
git checkout $CommitSha

foreach ($name in $skills) {
    & "$PSScriptRoot/vendor-skill.ps1" `
        -Source (Join-Path $tmp "skills/$name") `
        -SkillName $name `
        -UpstreamRepo 'obra/superpowers' `
        -UpstreamSha $CommitSha `
        -License 'MIT' `
        -Force:$Force
}
```

After running, the human still needs to (a) add `license: MIT` to each newly-vendored SKILL.md frontmatter (for Copilot CLI validator) and (b) hand-edit each UPSTREAM.md to match the richer M2 format. Both are minor — the script saves the bulk of the work.

Optional follow-up (not in scope for v1.1.0): teach `vendor-skill.ps1` itself to inject `license:` and to take a richer "My changes" parameter so this becomes one command. Note in the M4 follow-ups list.

## Drift risk and how to manage it

Claude Code users get fresh upstream via the marketplace; Copilot users get the snapshot from whatever commit SHA was vendored. Between releases, the two diverge. Acceptable because:

- Same upstream repo, same MIT license
- Snapshot doesn't go stale silently — `sync-superpowers.ps1` makes refresh a one-command job
- superpowers releases on a slow cadence (versioned, not main-tip)

Discipline: re-run `sync-superpowers.ps1` whenever a new superpowers release ships. Refresh PR is `chore: refresh superpowers snapshot to <SHA>` — version-bump rule will require a `[skip-bump]` token since we don't bump pitt-skills version for an upstream content refresh.

## Out of scope

- Vendoring superpowers' agents, commands, hooks, scripts (those go to Claude Code via the live marketplace; Copilot doesn't have analogues)
- Renaming superpowers skills' `name:` fields to add a prefix
- A general-purpose "vendor entire upstream marketplace" tool (only one upstream marketplace matters for now)
- Modifying superpowers' SKILL.md bodies to remove cross-references — leaving as-is is correct
